Confirmed 2026-06-14 via read-only prod query (Aldo Defilippis cancel case). The hourly `/jobs/process` → `runMailboxProcessing` correctly fetches, resolves provider, summarizes/classifies a thread, AND `identifyAndExtract` writes the `ClientEmailEvent` rows (legacy path) synchronously — but the **new registry proposal is frequently NOT created**, so nothing surfaces in the operator's "Client updates" panel and the client record never changes (e.g. a `permanent_cancel` left Aldo "active").

**Root cause:** in `src/services/roster-update-service.ts` (~line 271) the registry proposal draft runs in a `setImmediate(...)` fire-and-forget callback, itself inside an un-awaited `void postProcessThreadRoster(...)` (`src/services/thread-persistence.ts:281`). `ClientEmailEvent` + provider resolution persist (awaited, synchronous); the deferred proposal-draft callbacks don't reliably complete in the short-lived Cloud Run invocation, and failures are swallowed ("auto-draft registry proposal failed — skipping"). The manual `registry:backfill` script works because it awaits `createGmailClientRegistryProposal` to completion — that's why the only Aldo proposals in prod were 4 standing-fact drafts from a one-off backfill on 2026-06-04, all dismissed.

Note: the proposal path routes events to a client only on `matchClientName(...).kind === "exact"` (score ≥ 0.85, fuzzy/Levenshtein — so a 1-char misspelling like "Defillippis" still matches; that gate is NOT the cause here, but is a secondary silent-drop risk for low-score names).

**Scale (measured 2026-06-14):** NOT systemic — since the 06-04 backfill, 336 events vs 116 gmail_thread proposal sets across ~107 threads; only **12 (thread,client) pairs stranded** (events, no proposal), incl. 4 permanent_cancels (Aldo, Christine Armour, Berdecira Rivera) + holds/reinstates/schedule_changes. So the setImmediate auto-draft succeeds ~90% of the time — the 12 are sporadic deferred-draft losses (dropped/cut-off in the short-lived Cloud Run `/jobs/process` run), NOT a blanket failure.

**Confirmed fix direction:** running the awaited path `createGmailClientRegistryProposal(clientId, threadId, {confirmMismatch:true})` on the stranded threads re-extracted and drafted the correct `status_change/cancel` etc. proposals — proving extraction/routing are fine and only the deferred execution was lost. All 12 backfilled 2026-06-14 (drafts only, not applied).

**Secondary bug found:** 2 of the 12 (Ljerka Fenech `public_holiday_cancel`, Concetta Difilippo `new_service`) drafted an EMPTY proposal set — the per-client re-extraction in `createGmailClientRegistryProposal` (note/`getClientDraftInput` extractor) found nothing even though the thread-level `extractThreadRegistryItems` had produced an event. Divergence between the two extractors → some event types silently don't survive the per-client path.

**Why it matters:** ties to [[two-apply-systems-clientemailevent-vs-proposals]] (old event path still wins silently) and [[scheduled-inbox-processing]].

Different prompt → different extraction → the note prompt missed the `public_holiday_cancel` (Ljerka) and over/under-extracted `new_service` (Concetta) → empty proposal sets. NOTE: post-Bug-1 the normal pipeline is UNAFFECTED (it uses thread-level items directly); divergence only bites the manual UI button + backfill, plus a redundant 2nd LLM call. The mapper drops nothing; stale-drop (`dropStalePastOneOffServiceEvents`, grace=7d) did NOT cause Ljerka (6d past, within grace).

Secondary latent bug: `public_holiday_cancel`/`public_holiday_service`/`sw_no_show` are NOT in `BILLING_EVENT_TYPES` (`client-registry-llm-postprocess.ts`), so a >7-day-past PH cancel (scope point, dated) gets stale-dropped on reprocess even though it's invoice-relevant.
