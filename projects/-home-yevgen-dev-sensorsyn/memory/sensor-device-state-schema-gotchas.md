Querying prod device state for a property investigation (tbl_alarms in smokealarmprod, via runbook 07 SSM tunnel → 127.0.0.1:13308, `sensor-mysql` helper from load-prod-env.sh). Property key: tbl_properties.id (= CSV "id") or tbl_properties.GUID (= "BARN42/5"-style). Devices: tbl_alarms.propertyId. Hub vs children: alarmType in ("Sensor Hub","Sensor Smoke Alarm","Sensor Water Leak Alarm"); children point at the hub via controller/controllerId.

**Enum-ordinal trap (the big one):** `status` is `enum('0','1','2','3')` and `connectedStatus` is `enum('0'..'4')`. MySQL `WHERE status=1` matches the ENUM ORDINAL (position 1 = value '0'), so it silently returns the wrong/empty set. ALWAYS quote: `status='1'`. status '1'=active, '2'=removed/replaced (re-adds leave a ghost '2' row with the same serial, location NULL, lastTestDate NULL — ignore it; the '1' row is live).

**Columns that are NOT live-state (fleet-majority sticky flags — do not conclude "disconnected" from them):**
- hub `connectedStatus='0'` → 2696/3216 hubs. Default/unknown, not "offline".
- hub `disconnectionEmailSent=1` → 2535/3219. "we once emailed a disconnect", never cleared.

**Reliable liveness signal:** hub `lastConnectionTestDate` / `lastTestDate`. A nightly heartbeat cron (tbl_heartbeat_jobs, batched serials, executeAt ~00:08) drives a per-hub connection test; on a healthy day ~2500/3216 hubs stamp TODAY, ~170 yesterday, the rest are the genuinely-stale tail. So a today/yesterday timestamp = the hub really reported. Benchmark with `GROUP BY DATE(lastConnectionTestDate)` to place a hub in the healthy cohort vs the stale tail.

**Child alarms update lastConnectionTestDate RARELY** (only on scheduled tests, not daily) — only ~100/day fleet-wide. So a child stuck a few days back is NORMAL, not offline. Judge children by status='1' + connectedStatus='1' + testStatus=1 + recent property.lastTestDate, not by connection recency.

**Portal "Recommended Replacement within 1 Month" badge is BATTERY-driven, NOT expiry-driven — and false-positives on fresh installs.** `getAlarmDetail` (utils/Help.ts) decodes manufaturingDate from serial chars 16-20 (`20YY`+`MM`) and sets expireDate = +10yr, but that 10-yr expiry is NOT what lights the badge. ROOT CAUSE PINNED in the web portal `code/sensor-angular` (property-details/view/alarm component — the Hub&Products Battery column): `checkBattery(status,b1,b2){ return status>=parseInt(b1) && status<=parseInt(b2) }` (alarm.component.ts:232) with ranges 11-100=GOOD, 6-10=REPLACE_3_MNTS ("consider replacement within 3 months"), 0-5=REPLACE_1_MNTS ("RECOMMENDED REPLACEMENT WITHIN 1 MONTH"). When `batteryStatus IS NULL`, JS coerces `null>=0 && null<=5` → TRUE, so a device that simply hasn't reported battery renders the red "within 1 month" badge (plus a separate "-" from the `=== null` span). So the badge fires when batteryStatus is genuinely 0-5% OR NULL (unreported) — the NULL case is a frontend coercion bug, not a real low battery. NOTE the web portal has battery DISPLAY only — no battery-refresh button; the `{CMD:BATTERY}` refresh action ("Check Battery Status") exists ONLY in the mobile apps (iOS AlarmStatusVC / Android AlarmStatusFragment → PUT /users/alarms/check-battery). Fixing the null coercion = Category C change to original Sensor frontend code (regression-sensitive). A JUST-INSTALLED device commonly has batteryStatus=NULL for a cycle even though it's healthy (status='1', connectedStatus='1', testStatus=1, lowBattery=0, expireDate ~10yr out) → portal renders the null as a pessimistic "replace within 1 month". It does NOT appear in Alerts because it's a portal-side render of a NULL field, not a `tbl_alarm_alerts` event (no LOW_BATTERY row; lowBattery=0). Verdict pattern: decode the serial — if expireDate is years out AND lowBattery=0 AND no alert row, the badge is cosmetic. WHY a fresh install sits NULL & when it clears (per firmware/protocol doc `docs/upstream/extracted/technical-documents/SENSORGLOB-Firmware & Protocol Specification-110426-064630.txt`): child battery is PULL-only — the alarm reports battery ONLY when the hub is sent `{CMD:BATTERY}` (optionally `INDEX`). The 5-min heartbeat is connectivity-only (no battery); autonomous `LOW_BATTERY` (5/10/15% thresholds) exists but the subscriber early-returns on `CMD==LOW_BATTERY` (subscriber.ts:197). The backend issues the query as the tail of a `VERIFY→ALARMS→BATTERY` cascade (subscriber.ts:601-608), gated by `verifyEventTriggerSource` — and that cascade is SUPPRESSED during INSTALLATION (returns false when the alarm is in the install list, 5841), which is exactly why a just-installed child has batteryStatus=NULL. It populates on the next NON-install VERIFY cascade (the nightly HEARTBEAT connection-test allows the cascade), i.e. within the normal heartbeat cycle — NOT a month-away scheduled test. Operator force-lever: `checkAlarmBattery` endpoint (alarm.controller.ts:942) publishes `{CMD:BATTERY,INDEX}` for one device or `{CMD:BATTERY}` for all on the property — instant refresh. So: NOT a replacement; if it must clear now, hit checkAlarmBattery. Real case 2026-06-17: prop 16753 (11/3 Smythe Place, SMYT3/11) leak id 33919 serial 000000020121A0042406 → A004/2406 = mfd 2024-06, exp 2034-06, installed 2026-06-16, batteryStatus NULL → false badge, NOT a replacement. (Sibling leak 33917 same install read batteryStatus=100, no badge.) Compare batteryStatus across the property's devices to spot the lone NULL.

**Portal "Property ID" display ≠ raw GUID.** Portal showed "SMYTE/11" for what is GUID `SMYT3/11` (unit 11/3); the bundled-address line read like "1/3" but was 11/3. Match Haven units by `tbl_properties.GUID LIKE 'SMYT3/%'` and confirm by device set, not by the portal's prettified id.

**tbl_alarm_alerts** rows generated at the exact scheduled-test instant (DISCONNECT/BATTERY with eventStatus=0, createdAt == device lastTestDate) are test-cycle artifacts that resolve when the test passes (testStatus=1 stamped same second) — not a standing outage. eventStatus 0/1 is NOT a clean active/resolved flag (LOW_BATTERY is only ever 0). tbl_alarm_test_history / tbl_alarm_events_history are sparse/unused for Haven scheduled tests — don't rely on them being populated.

**Tracing comms actually sent to a tenant/agent (two complementary ledgers + a gate):**
- The per-property `tbl_properties.alertConfiguration` JSON GATES everything: per event-type (alert/tampered/disconnect/reconnect/low_battery/scheduled_test/...) × recipient (tenants/agent/agency/contractor) × channel (sms/push/email). Haven default seen: tenant gets SMS on alarm+tampered, SMS+email on scheduled_test; disconnect & low_battery = ALL false → no disconnect/low-battery comms ever leave, no matter the alert rows or sticky disconnectionEmailSent flag. Read this FIRST to explain "why no email".
- Email-template sends (property invite, install confirmation) → SendGrid; webhook events land in MySQL `tbl_sendgrid_activity` (email, event processed/delivered/open/bounce, template_id, createdAt). Query by `email=`.
- System event notifications (scheduled-test "about to occur", etc.) Email+SMS → logged in **MongoDB** `tbl_audit_histories` (fields: to.email, type Email|Sms, statusString SUCCESS/FAILURE, subject, createdAt) and `tbl_alarm_logs`. These go via SES, NOT SendGrid, so they do NOT appear in tbl_sendgrid_activity. Query mongo: `db.tbl_audit_histories.find({"to.email":...})`. Atlas IS reachable (IP allowlisted as of 2026-06) via `sensor-mongo`.
- App+cron logs are 365-day retention in CloudWatch: `smoke-api-prod-pm2-out-log` (stdout incl. SendGrid webhook payloads + Mongoose inserts), `-error-log`, `smoke-prod-api-cron-logs` (disconnect/test cron emails). Grep by serial / tenant email / GUID; time-box the window (full-range scan of pm2-out is slow).
- `timeout` can't wrap the `sensor-mysql`/`sensor-mongo` bash FUNCTIONS (not executables) — call them directly.

**Why a device event may produce NO SMS/email (the gating chain in `sendNotification`, sensor-alarm-backend `services/mqtt/subscriber.ts`):** ALERT (alarm) and TAMPERED both route through `checkAlarmAndScheduleNotification(payload, type) → sendNotification` (subscriber.ts:407–444, 830–848). `sendNotification` (subscriber.ts:993–1025) sets `caseType = payload.type.toLowerCase()` (e.g. "alert"/"tampered"/"low_battery") and sends a channel ONLY where `JSON.parse(property.alertConfiguration)[caseType][recipient][0][channel] === true`. So an actual fire/leak ALARM silently notifies NOBODY if the property's `alert` gate is all-false — NOT a code bug, it's the per-property config (the portal Configuration tab → "Alert" rows; unticking them silences real alarms with no warning). Extra gates: sends only if there's a COMPLETED/CLOSED install job (`if (!installJob) return`), only when property ACTIVE, and a flapping-suppression filter (`FLAPPING_ALERT_CONFIG`) drops rapidly toggling alerts. The default `DEFAULT_PROPERTY_CONFIGURTION` (constants/app.ts:1300) has `alert→tenants.sms=true` (a new property DOES SMS the tenant on alarm); the portal RESET restores that default. Live MQTT decode for alarm events: `{CMD:ALERT,INDEX,ALARMTYPE,STATE}` STATE 1=on 2=off; ALARMTYPE 1=smoke, 3=water leak; also `{CMD:HUSH,...}`, `{CMD:TAMPERED,INDEX,STATE}`, `{CMD:POWER,POWERSTATE}` (2=mains lost/on-backup→logged as hub BATTERY alert, 1=restored). Comm `tbl_audit_histories` docs: Alert/Sms docs record `to`, but template Emails (install confirmation) carry `to:{}`/`{name:'Audit Log'}` (recipient not logged). Verified live 2026-06-18 on rehearsal prop 23796 (26 Kardinia St): alarms didn't notify because "Alert" was unticked mid-test; tamper/hush did. See [[saferops-test-accounts-and-duplicates]].

**Property "Disconnected" while devices "Passed" = stale `status` from an uncompleted job (NOT an outage).** `tbl_properties.status` (PROPERTY_STATUS enum: ACTIVE='1', DISCONNECTED='10', quote it!) and device liveness are INDEPENDENT state machines. Hub removal/swap (`alarms.controller.ts` removeController/unallocateController/removeControllerForTestPack) sets property `{status:'10', installedAt:null}` AND wipes alerts (`AlarmAlerts.update({eventStatus:false})`) — so there's NO row in Alerts & Management to explain it. The reset to ACTIVE('1') (+ installedAt/alarmTestDate) fires in EXACTLY ONE place: job completion (`jobs.controller.ts:2392`). So a swap whose hardware is done (new hub+children test Passed) but whose job is left `status=ACCEPTED(2)`/`completionDate=NULL` stays stuck DISCONNECTED forever with healthy devices and an empty Alerts page. Triage: property `status='10'` + hub recent lastConnectionTestDate + children testStatus=1 → stale uncompleted job (find it: jobs `status` ACCEPTED/PENDING, completionDate NULL). Fix = COMPLETE the job via normal flow (NOT a raw `UPDATE tbl_properties.status` — that hides the open job + null installedAt). BUT verify the hub really tested: a property whose hub is `connectedStatus=0`/`testStatus=0` with a months-old ACCEPTED job is a GENUINE incomplete install (coverage gap), not a cosmetic flag. Cases 2026-06-17 (agency 37413): TUCK49A/22073 job 6388 & MOOR322/203/16733 job 6409 = stale-swap (cosmetic); NEAL207/1/20552 job 4266 since April = real non-install. Logged as gap #23 (`docs/gaps-detailed.md`).

**Hub re-pair/swap creates a NEW "generation" of child rows; the old rows ghost out.** On a re-pair (jobId stamped), the live devices get NEW tbl_alarms ids with `propertyId` set + `jobId` set + today's installationDate + status='1'; the prior generation flips to status='2', connectedStatus='2', `propertyId=NULL`, jobId NULL (children re-bond by SAME serialNumber across generations). Pairing churn also leaves transient status='2' rows with location=NULL/testStatus=NULL — ignore. **A device that FAILED to re-pair simply has NO status='1' row** (only its old status='2' ghost) — that's the coverage-gap signal, not a DB error. `testStatus` is a tinyint (NOT the status enum): 1=Passed; **2 = portal "In progress…"** (test initiated, no PASS recorded — e.g. TEST horn fired downlink but the uplink VERIFY/result was lost, classic QoS0 / post-pairing test-timing failure). Case 2026-06-17: STEW49/20 (prop 21700, "20/49 Stewart Drive") hub re-paired under completed job 6420 — flagged "1st Floor Hallway" smoke `000000010384A0012405` (id 30723) hit testStatus=2 ("failed no sound"/stuck) and did NOT re-pair (no status='1' row); the re-pair swapped in a different smoke `000000002620A0012503` ("upstairs hallway") and property went ACTIVE/all-Passed. Operator follow-up for such cases: confirm the dropped position physically has a working (re-paired/replacement) alarm, not a silent coverage gap.

**GENUINE offline hub (the REAL outage, vs the two cosmetic cases above) — the signature:** hub `status='1'` but `connectedStatus='0'` AND `testStatus=0`, with `lastConnectionTestDate` stamped recently (the nightly heartbeat cron records when it ATTEMPTED — NOT a hub response; the 0/0 is the failed result) and an **active `tbl_alarm_alerts` DISCONNECT row with `eventStatus=1`** whose `updatedAt` is bumped each failed test + `disconnectionEmailSent=1`. The hub's `batteryStatus`/`lowBattery` are the LAST values it reported before dropping (stale — e.g. battery 97% but `lowBattery=1` = mains lost / on backup right before it went dark). Children show portal "Failed" (testStatus=2, lastConnectionTestDate frozen weeks back) because the dead hub relays for them. **Property stays `status='1'` ACTIVE** — a hub merely going offline does NOT change property.status (only explicit removeController does), so Property & People reads "Active" while the hardware is down (OPPOSITE of the stale-job case). Portal UX combo "green Battery dot + red Status + Failed" = stale battery + dead link. This needs a FIELD action (restore mains/replace hub/SIM), not a DB fix; confirm true-offline via CloudWatch `payload-># <hubSerial>` (inbound ground truth). Case 2026-06-17: STEW49/10 (prop 21689, hub 7215 `003320137293C0012406` "Garage Under NBN") offline since ~late-March (children last real contact 2026-03-26), disconnect alert id 8856 eventStatus=1 active + emailed to agent — a genuine ~3-month outage. So the THREE swap/disconnect patterns: (1) stale `status=10` from uncompleted job = cosmetic, complete the job; (2) re-pair ghost generation / testStatus=2 stuck device = usually resolved by the re-pair, check coverage; (3) connectedStatus=0/testStatus=0 + eventStatus=1 disconnect = REAL offline hub, field action.
