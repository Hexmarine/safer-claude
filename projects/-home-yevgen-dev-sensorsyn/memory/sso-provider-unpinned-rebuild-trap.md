**The IdP `sso-provider` (auth.sensorglobal.com, the OIDC provider behind every Sensor + safer-ops login) has a build trap: builds are UNPINNED and prod had not been rebuilt in ~10 months.** `package-lock.json` + `yarn.lock` were **deleted and gitignored** in `SENS-6440` (Jul 2025); prod ran a **Jul-2025 artifact** the whole time. The local `package-lock.json` on disk is untracked → NOT in the CodeBuild build. So `npm install` at build time resolves *latest* matching versions (e.g. `oidc-provider "^7.14.3"` → latest 7.x + all transitive drift).

**2026-05-31 outage (ROOT CAUSE CONFIRMED + FIXED):** a 1-line commit (`claimsParameter` enable, [[saferops-idtoken-email-claim-gap]]) was pushed → **triggered the first CodeBuild rebuild in ~10 months** → unpinned `npm install` pulled a **newer mongoose**, whose `autoIndex` finally **built a latent-buggy index** that had sat dormant in `src/db/mongodb/models/BaseModel.ts` since 2023: a **`unique` index on `payload.grantId`** (`payload.grantId_1`) over AccessToken/AuthorizationCode/RefreshToken. oidc-provider intentionally gives the auth code + access token (+ refresh token) the **same `grantId`**, so once that unique index exists every `/token` exchange's AccessToken write hits **E11000 duplicate key → 500** → **ALL logins broke**: safer-ops code-grant returned `{"statusCode":500,"message":"OIDC token exchange failed: 500"}`; trader/password-grant showed "oops! something went wrong". The 500 was **never logged** — oidc-provider surfaces issuance failures as a `server_error` event and this app attaches **no error-event logger** (traceability gap). The one-liner only *triggered the rebuild*; it wasn't the breakage.

**Why the CodeDeploy rollback DID NOT fix it:** the index lives in **MongoDB**, not the build — rolling the code back to the July artifact left the index in place, so 5xx continued. **The actual fix was dropping the Mongo index** (login recovered instantly, LB 5xx → 0): `sensor-mongo --quiet --eval 'db.getSiblingDB("sensorproddb").basemodels.dropIndex("payload.grantId_1")'`. The old build's mongoose hadn't rebuilt it for 10 months, so the drop sticks on the current fleet. **Permanent source fix: remove `unique: true` from the `payload.grantId` index in `BaseModel.ts`** (it must never be unique) — otherwise a future boot with a newer mongoose recreates it.

**IdP Mongo = Atlas** `sensor-prod.yujwyg.mongodb.net`, db `sensorproddb`, collection `basemodels` (oidc-provider Grant/Session/AccessToken/AuthorizationCode/RefreshToken, keyed by `key`, `payload.kind`). Creds are in the `sensor-prod` secret (`MONGO_DB_URL`) — `scripts/load-prod-env.sh` now exports `MONGO_DB_URL`/`MONGO_DB_DATABASE` and defines a **`sensor-mongo`** helper (mirrors `sensor-mysql`). Atlas **IP-allowlists** (add your egress IP in Network Access or mongosh times out) and the user is **db-scoped** (no `listDatabases`/`config` — target `sensorproddb` directly, don't scan).

**Deploy/infra facts:** CodePipeline → CodeDeploy **blue/green** on ASG, compute platform Server. App `sso-api`, deployment group `sso-api-dg` (region ap-southeast-2, profile sensorsyn-mfa). Log groups `sso-prod-api-logs` / `sso-prod-api-error` (pm2→CloudWatch). NOT in k8s (the safer-ops EKS cluster has no sso pods). Prod access is post-JV; the pipeline debt is inherited from the prior owner.

**Rollback procedure (what fixed it):** redeploy the previous good deployment's S3 artifact via `aws deploy create-deployment --application-name sso-api --deployment-group-name sso-api-dg --revision '{"revisionType":"S3","s3Location":{...prev artifact...}}'`. Get the prior revision from `aws deploy get-deployment --deployment-id <prev>` (the good one was `d-1L0UPX5BI` / artifact key `sso-api/BuildArtif/TpCw7L7`). Blue/green is SAFE to roll back: a failed green fleet doesn't drop traffic from the current fleet. Verify success by `claims_parameter_supported` flipping back in `/.well-known/openid-configuration` (was the only behavioural marker of the bad build). Console alt: CodeDeploy → sso-api → sso-api-dg → prior deployment → Retry. Agent cannot run `create-deployment` itself (prod-deploy classifier blocks it) — hand the command to the user.

**RULE going forward:** never trigger an sso-provider prod rebuild without (1) committing a pinned `package-lock.json` (un-gitignore it) so builds are reproducible, and (2) verifying a real login on uat/sandbox (`uat-sso-codedeploy` / `qa-sso-api`) first. The `claimsParameter` + safer-ops email-claim fix must ride a pinned, sandbox-verified build — do not re-push to main until then or the pipeline redeploys the broken `vimVFmq`. See [[saferops-idtoken-email-claim-gap]], [[saferops-live-trace-rig]].
