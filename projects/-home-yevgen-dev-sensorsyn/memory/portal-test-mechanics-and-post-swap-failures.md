How the Sensor portal "Test All" works (`testAlarm`, alarms.controller.ts): publishes `{"CMD":"VERIFY"}` to `sg/sas/cmd/<hubSerial>`, then `{"CMD":"TEST"}` 5s later via in-process setTimeout, both **QoS 0 fire-and-forget**; a 360 000 ms setTimeout (`checkAfterSixMin`) marks anything still in-progress as Failed / "Hub did not respond". Failed-device signature in `tbl_alarms`: `startTestDate` and `lastTestDate` exactly 6 min apart, `testStatus=2`. The hub's own row passing (`lastConnectionTestDate` fresh) while devices fail = hub answered VERIFY but never executed/answered TEST.

Ground truth for "what did the hub actually send": CloudWatch `smoke-api-prod-pm2-out-log`, filter `'"payload >" "<hubSerial>"'` — subscriber.ts:195 logs EVERY inbound message on the shared subscription. TEST commands are deliberately NOT logged outbound (`addMqttCommandsLogCron` returns early on CMD==TEST). The Mongo `tbl_alarm_logs` "Hub connection status verified" entry lands exactly publish+60s (delayed bulk-create from Redis), not at response time.

**Case 2026-06-11 (BAND26, prop 21727, hub 001621661832C0012406):** hub dead on old SIM for days (daily 00:38Z VERIFY failures), Onomondo swap → boot 22:00:44Z, ops triggered test 22:02:40Z (~2 min after boot). Hub answered VERIFY (battery 91, AC) but ignored TEST published <1s later; alarms never sounded → 6-min timeout fail. Cause class: tested too soon after boot — either QoS0 drop on the freshly-attached cellular session or RF link to paired alarms not yet re-established. Nothing actually broken; retest later passes.

**Counter-case same day (TUCK49A, prop 22073, hub 003674087621C0012406):** identical prologue (hub dead ≥Jun 8, swap, boot 00:27:30Z, test +3min) but hub DID execute the test: Entry smoke passed in 19s, Sink water-leak passed at the 5-min mark (RF retries are slow!), Hallway smoke FAILED via explicit hub-reported `{"CMD":"TEST","INDEX":2,"STATUS":2}` at +7.5min — genuine RF/device fault (was already connectedStatus=0), needs site visit. **The two signatures (REFINED — see correction below):** all devices fail together at exactly start+6:00 with zero inbound TEST responses = command lost/timeout, retest fixes; device-specific STATUS:2 from hub = the hub executed the test and got a real reply. Note: hub retries past the portal's 6-min window and a late STATUS:2 still lands (the in-process checkAfterSixMin timer doesn't always fire — lost on instance recycle). BATTERY poll answers can be hub-cached — a "disconnected" device can still report 100%.

**CORRECTION 2026-06-15 — "explicit hub STATUS:2 = real fault" is NOT absolute.** A hub-reported STATUS:2 can itself be a transient post-reboot settling artifact that clears on retest. Case BAND15A (prop 21723, hub 000019514859C0012406): SIM swap → POWER AC 04:46Z → first test TEST40409 at 04:48Z (~2min after boot) reported Hallway smoke (idx0) `{"CMD":"TEST","INDEX":0,"STATUS":2,"eventTriggerSource":"HUB"}` — explicit hub failure, NOT a timeout (all 3 devices reported same millisecond, no "did not respond" doc). Retest TEST92557 at 04:56Z (~10min after boot) → idx0 STATUS:1 PASS. Device stayed connectedStatus=1, battery 100. **So timeout-vs-STATUS:2 is NOT the fault discriminator.** The real discriminator = *persistence across a retest after the hub has settled* + connectedStatus: genuine fault (WATT29 still connected but STATUS:2 holds; WATT23B STATUS:2 then dropped to connectedStatus=0) survives retest; transient (BAND15A, also TUCK49A's slow-RF passes) clears. A single STATUS:2 within ~2-3 min of boot is inconclusive — always retest after the 5-10 min settle window before condemning.

**Third signature — phantom duplicate rows ("looks failing", isn't):** re-pairing during install leaves orphan `tbl_alarms` rows for the SAME serial (no `indexNumber`, no `location`, `connectedStatus=2`, `testStatus` NULL, frozen `updatedAt`) alongside the real bound row (has index, `connectedStatus=1`, passes). The ghost renders red/disconnected forever, so a serial lookup looks like a failing device. This is the "ADD STATUS 2 already exists" re-pair residue (see [[prepared-kit-backend-internals-doc]]) surfacing in the portal. Disambiguate by `indexNumber` + `connectedStatus`, not by serial. **Case 2026-06-15 BARN14/1 (prop 21730):** water leak `000000020211A0042406` flagged "failing" but the indexed row (id 31794) PASSED test TEST83851 same day; three orphan rows (ids 31791/31792/31793, created Feb-03, frozen Mar-25) were the ghosts — cleanup = deactivate those rows, NO site visit. WATT23B (22089) had the same residue (orphan smoke row 31491) **plus** a genuine water-leak fault — the two can coexist. **Taxonomy of "water leak failing" 2026-06 batch:** WATT29 (22093)=real (hub STATUS:2, still connected); WATT23B=real (STATUS:2, since dropped to connectedStatus=0) + orphan; BARN14/1=no fault, pure ghost. Ground truth always = `tbl_alarm_logs` hub report + indexed row `testStatus`, never raw serial count.

**Ops guidance:** after SIM swap / power cycle, wait 5–10 min after solid blue (or run the silent hub-connection-test first) before triggering a sounding device test. Live-watch a retest with `./scripts/mqtt-listen.sh <hubSerial>` over the MQTT tunnel. A manual VERIFY publish to `sg/sas/cmd/<sn>` triggers the backend's chained ALARMS+BATTERY follow-ups = full silent health snapshot (user-approved op, 2026-06-12).
