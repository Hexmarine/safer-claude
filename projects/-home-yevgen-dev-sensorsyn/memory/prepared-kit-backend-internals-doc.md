`safer-ops/docs/investigations/2026-05-24-prepared-kit-backend-internals.md` — the internals + debug-playbook doc for the depot→installer prepared-kit flow (pair-stock / stock-verification / adopt-on-attach / release-stock). Read it before diagnosing any pair / verify / attach / release issue. Contract companion: `safer-ops/docs/prepared-kit-sensor-api.md`.

The key non-obvious model it captures: a child alarm is bound to a hub in **two independent places** — the `tbl_alarms` DB row (`controllerId`/`connectedStatus`) AND the hub's firmware ALARMS list — and they can diverge. Release must clear **both** (soft-delete the DB row *and* MQTT `{"CMD":"REMOVE"}`) or the serial isn't truly reusable.

**The classic stuck-pairing failure (root-caused 2026-05-28 against the firmware spec):** firmware still holds a child while the DB row was soft-deleted/FAILED → a re-`ADD` returns **`STATUS 2`**, which the **Firmware & Protocol Spec defines as "AlarmSerial already exists"** (`docs/upstream/Technical Documents/Firmware+&+Protocol+Specification.doc`: ADD BoolStatus 0=Undefined,1=Success,2=Failure/"already exists"; **there is NO STATUS 3** — the backend's STATUS-3 branch is dead code; the real "already bonded" reply is **STATUS 2**). The native subscriber treats STATUS 2 as a no-op, so the row stays IN_PROGRESS and the 10s `failAddAlarm` timer flips it to **FAILED (`connectedStatus=4`)**; the next ADD STATUS-1 handler sees FAILED → publishes REMOVE → DELETED → re-ADD → STATUS 2 → … endless ADD↔REMOVE churn + duplicate `tbl_alarms` rows (saw 17 for one serial). safer-ops reads `connectedStatus` via `status-by-serial`, which never reaches `1` in this state → pairing never confirms.

**ENUM DISCREPANCY (important):** the in-code `ADD_EVENT_STATUS` (constants/app.ts:1908) maps `2=FAILURE, 3=ALREADY_EXISTS` — but that was never validated against hardware. The **firmware spec + live MQTT both show STATUS 2 = "already exists"**, and the hub **never emits STATUS 3** (its handler is dead). Trust the spec, not the enum.

**Isolated fix built 2026-05-28 (sensor-alarm-backend, gated to the depot STOCK flow ONLY — native install byte-identical; codex-clean after 3 rounds):** `pairStock` **awaits** setting a per-(hub,alarm) Redis flag `STOCK_PAIR_<hub>_<serial>` (~30s TTL) BEFORE publishing ADD (so a fast reply can't beat the flag). The global MQTT ADD handler, via `reconcileStockPairAlreadyBonded` (subscriber.ts), handles **both STATUS 2 and 3** (defensive): flag present → `alarmEntity.reconcileStockAlarmConnected({serialNumber, controllerId})` flips the in-flight STOCK (propertyId NULL), ACTIVE, IN_PROGRESS/FAILED row **bound to the responding hub's controllerId** → CONNECTED (timer can't strand it; the VERIFY→ALARMS roster sync is the truth backstop — fills index, or soft-deletes a row not actually rostered). STATUS 1 on a FAILED row → REMOVE only when NOT a stock pair (`shouldRemoveFailedAlarmOnAdd`). Isolation logic is the pure `src/utils/stockPairing.ts` (unit-tested `test/alarmStockPair.unit.test.ts`, 4 tests; 51 backend unit tests green, typecheck+lint clean). Native ADDs never set the flag → both branches skipped. **Shipped 2026-05-28 (the STATUS-2 reconcile) — separately deployed; the depot pairing now converges.**

**Stock-only RESET clean-slate (#105, 2026-05-28, codex-clean; committed since, verified 2026-07-03):** new `POST /users/alarms/reset-stock-controller` → `alarmEntity.resetStockController` is the depot "Reset" path for a not-yet-installed kit. It factory-resets the hub via the firmware **`{"CMD":"RESET"}`** (unlinks ALL its alarms in ONE command — the firmware spec's RESET = "reset controller to factory settings, unlinking all linked alarms"), so a churned/stuck stock pairing is wiped clean instead of per-child REMOVE. STOCK-ONLY and **does NOT touch the SIM or any property/job** (unlike `removeController`, which is the installed-hub decommission and deactivates the SIM). Concurrency-hardened over 5 codex rounds: the entity runs in ONE transaction that locks the hub row **FOR UPDATE**, re-checks `propertyId IS NULL` under the lock, **refuses if `getBlockingOperationByControllerSerial` finds an in-flight attach op** (AWAITING_VERIFY/ALARMS, MATCHED, or ATTACHING — covers the whole attach commit window) → 409 `ATTACH_IN_PROGRESS`, soft-deletes the stock children, and publishes the RESET **inside the lock** (fire-and-forget MQTT). No post-reset op-cancel (it'd kill a fresh op a concurrent attach legitimately started after the lock released). safer-ops `resetKit` calls it for stock kits (`!kit.propertyId`), falling back to `releaseKitDevicesOnSensor(forceByDeviceId:true)` if the hub isn't stock / RESET fails. **Reset + "Return hub to stock" are now ONE action** — see [[stranded-device-detach-recovery]].

Also in the doc: the ADD/REMOVE STATUS code tables, device-operations model (30-min TTL, type/property-agnostic handlers, the audit `_events` table), the backward-compat invariants that keep native install/replace byte-identical (the `adoptPrePaired` flag), and the debug playbook ("when X breaks, look here") + live access (MySQL `smokealarmprod` tunnel localhost:13308; test accounts agency.test=59202 / trade.test=59203).
