How the gating works (traced when it was OFF):** all tenant SMS funnels through `sendOtp()` in `code/sensor-alarm-backend/src/libs/smsManager.ts`. Two nested env gates: master `process.env.SEND_SMS == '1'` (line 92), then allowlist `SEND_SMS_ONLY_ON_ALLOWED_LIST == '1'` (line 93, DB list `tbl_sms_allowed_list` via `getSMSAllowedListToValidate()` where `status='1'`; a number not on the list **silently drops — no `else`**; match is **exact string** `phoneCode+phone`, no normalisation → format must mirror the tenant's stored values).
- **SOURCE OF TRUTH = `sensor-prod` Secrets Manager secret, NOT the on-box `.env`.** Boot order (`index.ts`): `dotenv.config()` loads a tiny `.env` (AWS creds + `SECRET_NAME=sensor-prod`), then `src/libs/secretManager.ts getSecretByARN()` fetches the secret and `Object.assign(process.env, secret)` overlays ALL config. So `/proc/environ` AND the on-box `.env` are BOTH blind to it — earlier `.env`/`/proc` reads were a dead end. To read/check any live config value, read the SECRET (or `load_env`).

**Failed-SMS report feasibility:** the hard part already exists — SNS **delivery-status logging is enabled** in ap-southeast-2 (`DeliveryStatusSuccessSamplingRate=100`, role `SNSSuccessFeedback`, senderID `SENSOR`, $700/mo cap). Two CloudWatch log groups, **365-day retention**: `sns/ap-southeast-2/747293622182/DirectPublishToPhoneNumber` (success) + `.../Failure` (carries `status`, `providerResponse`, `phoneCarrier`, `mnc`, `messageId`, destination). So true handset non-delivery IS observable. See [[daily-email-volume-overdue-job-reminders]].
- **Gap:** SNS logs key on phone+messageId, not property. App **discards the SNS MessageId** (`smsManager.ts:60` logs+throws it away) → no join back to property/tenant. To build the report: either persist MessageId at send time and join on it, or correlate failure-log phone+timestamp against the 24h cron's recipients (fuzzier).
- **Caveat for any report:** the silent-drop gate means "suppressed by gate" and "never attempted" both look identical to "0 failures." Don't ship a report that reads empty because SMS is simply off.

**Bottom line for the customer:** before any failure report is meaningful, SMS must first be turned on for tenants (set `SEND_SMS=1` + decide allowlist on/off — a config + approval question, owner of the `.env`/secret). Until then the test SMS is dark, so home-phone outreach is currently the ONLY tenant notification happening.
