**Why it was needed:** `logToConsole` (utils/Help.ts, ~938 calls) is gated to local/dev/qa — a no-op in prod. No access log, no structured/correlated lines. **Sentry is configured but the team has NO access to the project**, so captured exceptions are invisible — not a usable safety net.

**The sink decision:** backend runs PM2-on-EC2; stdout already ships to CloudWatch group `smoke-api-prod-pm2-out-log` (also `-error-log`, `smoke-prod-api-error-logs`, `-cron-logs`). Verified LIVE 2026-06-16 (prod stdout flowing now; Mongoose debug + KORE/Twilio SIM dumps, which also leak imei/imsi/iccid → justifies redaction). So **CloudWatch Logs Insights = our self-owned Sentry replacement** (`filter level="error"`), no SaaS, no infra change. Multi-line pretty dumps currently fragment across CW events; pino's one-line-per-event fixes queryability.

**What shipped (Phase 1, additive A/B):** `src/utils/structuredLogger.ts` (pino 10, LOG_LEVEL env, central secret redaction + custom `safeErrSerializer` that strips axios err.config/request/response so bearer tokens & response bodies never reach logs, `createLogger()` factory for tests), `src/utils/requestContext.ts` (AsyncLocalStorage + `getLogger()` child), `src/middlewares/observability.ts` (requestContextMiddleware reuses [[traceability-rule-saferops]]'s `resolveCorrelationId`, echoes `x-correlation-id` response header; accessLogMiddleware one JSON line/req, skips `/api/v1/monitoring/health-check`, userId from `req.userDetail`, aborts→warn). Edits: index.ts (mount mw after cookieParser; structured error log in fallthrough handler w/ header-fallback correlationId for body-parser errors; process unhandledRejection+uncaughtException handlers are **LOG-ONLY, no exit**). properties.entity.ts processUploadProperties catch logs propertyFileId/agencyId before the existing rethrow (the [[csv-import-email-throttle-aborts-finalize]] incident). 17 unit tests pass, tsc clean.

**Process-handler gotcha (corrected after xhigh /code-review):** do NOT call `process.exit(1)` in app-level unhandledRejection/uncaughtException handlers here. Sentry's default integrations already own lifecycle — OnUnhandledRejection runs mode 'warn' (logs, process SURVIVES today), OnUncaughtException captures+flushes+exits. exit(1) would (a) turn the cosmetic CSV-454 throttle into a full crash+PM2 restart and (b) pre-empt Sentry's async flush. codex's bare `node -e` test missed this (no Sentry loaded); only the deeper review caught it. Log-only is the behaviour-preserving choice.
