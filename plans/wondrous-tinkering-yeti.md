# Deploy & rollback — Phase 0 of the leaked-OIDC-key remediation

## Context

The remediation code that externalizes the committed OIDC/JWT signing keys is built and
reviewed (codex-clean). We are deploying **Phase 0 only**: externalize the keys to the
secret stores with each service's **current key preserved**, so behavior is byte-identical
and nothing rotates yet. Rotating/retiring the leaked key is a deliberate later window
(runbook kept at the end for reference). User has approved the push.

State of the code:
- `sensor-alarm-backend`: committed as `ef229a60e` on branch `fix/remove-leaked-sso`
  (TokenManager signs with active key + stamps `keyid`; `auth.ts`/socket/`verifyToken`
  resolve by header `kid` with `SSO_SID` fallback; dead `rsa.key` reads + files removed).
- `sso-provider`: **uncommitted on `main`** (`jwkKeys`/`cookieKeys` externalized to
  `SSO_SIGNING_JWKS`/`SSO_COOKIE_KEYS`, verify-across-all-keys, fail-fast in non-dev).

## Deploy-critical facts (established by review)

- **Runtime:** both services run on **EC2 + pm2 via AWS CodeDeploy** (`appspec.yml`
  → `ApplicationStart` → `scripts/start-server.sh`). The k8s manifests under
  `code/infra/k8s` are **local-dev only** — not prod. Provider start script ships a
  **prebuilt `build/`** and does **not** `npm install` at boot.
- **Secret merge:** both bootstraps do `Object.assign(process.env, secretObject)`
  (`backend/src/libs/secretManager.ts:25`, `sso-provider/src/utils/secretManager.ts:24`),
  so any key added to the secret JSON appears as an env var. Backend secret = `sensor-prod`
  (`SECRET_NAME`); provider secret = `sensor-prod-sso`.
- **Each service signs with a DIFFERENT current key** (NODE_ENV split):
  - provider `NODE_ENV=production` → active+published OIDC key = `vXYg…`
  - backend `NODE_ENV=dev` → `TokenManager`'s own reset/folder/login tokens = `sqy3…`
    (the backend's `production`/`vXYg` block was **dead code**, never selected)
  - provider cookie key = `"subzero"`
- **Asymmetric failure-on-missing-secret:** provider (`production`, not in DEV_ENVS) →
  **fail-fast, won't boot** if `SSO_SIGNING_JWKS`/`SSO_COOKIE_KEYS` absent. Backend
  (`dev`, in DEV_ENVS) → **silently loads the throwaway dev key and breaks token flows**
  (no crash). ⇒ **populate the secret before the code deploys, and verify.**
- **`SSO_SID` (backend) is independent of `SSO_SIGNING_JWKS`:** `SSO_SID` is the JWKS
  lookup fallback → points at the **provider** key (`vXYg` thumbprint). `SSO_SIGNING_JWKS`
  (backend) is `TokenManager`'s **own** key (`sqy3`). Do **not** conflate them.

> AWS Secrets Manager writes are mutations → require an **approved elevated mutation
> window** before Step 1. CodeDeploy deploys require an **approved deployment window**.
> Never print secret values — verify by shape (present / length / `JSON.parse` ok / array
> length). User performs all git commits/pushes.

## Step 0 — Pre-flight (read-only)

1. Confirm live `NODE_ENV` + `SECRET_NAME` per box (pm2 env / SSM; needs a diagnostic
   window if via SSM). Verify backend really runs `dev`, provider non-dev, and the IAM
   role can read each secret.
2. **Capture current JWKS:** `GET ${OIDC_ISSUER}/jwks` → record the current `kid`(s) (the
   `vXYg` thumbprint). In-flight session tokens carry this; Phase 0 must keep publishing
   the same `kid`.
3. **Provider rebuild-trap pre-check** (the main provider risk — see
   `sso-provider-unpinned-rebuild-trap`): the shipped artifact's
   `node_modules/mongoose` must equal the committed lockfile (`8.9.5`), `oidc-provider`
   must match lockfile, the `BaseModel.ts` no-`unique:true` fix from the prior incident
   must be present, and the build must use `npm ci` (caret deps + first rebuild in months
   = latent unique index on `payload.grantId` → E11000 → all logins 500).
4. Confirm `scripts/gen-jwks.js` exists where each `.env.example` references it; if not,
   keys for later phases can be generated by a one-off node script.

## Step 1 — Populate secrets FIRST (elevated mutation window)

Add to **`sensor-prod-sso`** (provider):
- `SSO_SIGNING_JWKS` = JSON array, sole element = current provider key `vXYg…`,
  **no explicit `kid`** (so oidc-provider derives the identical thumbprint → published
  JWKS `kid` unchanged → in-flight tokens keep verifying).
- `SSO_COOKIE_KEYS` = `["subzero"]` (current value preserved).

Add to **`sensor-prod`** (backend):
- `SSO_SIGNING_JWKS` = JSON array, sole element = current backend TokenManager key
  `sqy3…`, **no explicit `kid`** (backend keeps minting kid-less tokens as today;
  `verifyToken` tries-all).
- **Leave `SSO_SID` unchanged** (provider `vXYg` thumbprint).

Secrets are read only at boot, so populating now is inert until the new code deploys (old
code ignores these keys). Verify by shape only.

## Step 3 — Deploy backend first, then provider (deployment window)

Order: backend first (more forgiving; verifies bearer via JWKS) while the provider still
serves the same `vXYg`/JWKS; then the higher-risk provider rebuild.

1. Deploy backend revision. Boot check: pm2 up, no `SSO_SIGNING_JWKS`/JSON errors, and
   **confirm it did not silently use the dev fallback** — exercise a backend-minted flow
   (password-reset verify or folder verify) on a fresh + an in-flight token.
2. Deploy provider revision. **Watch the first rebuild:** tail
   `/var/log/pm2/sso-api-dev-error.log` for `E11000` / index build on `payload.grantId`.
   If it appears → all logins 500 → execute provider rollback (below).
3. Provider boot check: `GET ${OIDC_ISSUER}/jwks` returns the **same `kid`** captured in
   Step 0; `/.well-known/openid-configuration` OK.

## Step 4 — End-to-end verification

1. **Login/session:** `getTokenViaPassword` (Haven login) → `id_token`; call an authed
   endpoint with `access_token: bearer …` → `AdminUserAuth` accepts (header-`kid` path).
2. **OIDC exchange:** safer-ops depot/installer login → `sensor-exchange` →
   `/users/sso/exchange` succeeds.
3. **Socket auth:** socket login path accepts the token.
4. **Password reset:** mint `FORGOT_PASSWORD` → `reset-password-with-otp` → `verifyToken`
   accepts (multi-key verify; in-flight `sqy3` tokens still valid).
5. **JWKS shape:** single key, `kid` unchanged from Step 0.
6. No new errors in either pm2 log over a watch window.

## Rollback

Phase 0 rollback is **clean** because keys are unchanged:
- **Code rollback:** redeploy the previous CodeDeploy revision for the affected service.
  Old code uses its hardcoded keys — byte-identical to what's now in the secret → no token
  breakage. The added secret keys are harmless to old code (ignored). Keep them.
- **Provider rebuild-trap:** if `E11000` on `payload.grantId` → redeploy previous provider
  revision **and drop the offending Mongo index in Atlas** (rollback alone won't remove it
  — it's Atlas state), per `sso-provider-unpinned-rebuild-trap`.
- **Backend silent-dev-key symptom** (reset/folder flows fail / "invalid token"): secret
  wasn't picked up → fix secret/`SECRET_NAME`, restart pm2 or redeploy.
- ⚠️ Clean code-rollback holds **only at Phase 0**. Once a later window promotes a new
  signing key, code-rollback is unsafe — rollback then = **secret reorder only**.

## Follow-on rotation (separate window — NOT this push)

After Phase 0 is verified stable, retire the leaked key gracefully:
1. **Add new key (verify-only):** generate fresh RSA JWK (explicit `kid`,
   `node scripts/gen-jwks.js`); append as 2nd element of `SSO_SIGNING_JWKS` in **both**
   secrets; redeploy both. JWKS publishes both; nothing signs with the new key yet.
2. **Promote:** reorder so the new key is element [0]; redeploy. New tokens sign with the
   new key; old-key tokens still verify. Update backend `SSO_SID` → new `kid`.
3. **Retire leaked key:** after a **short** grace window (hours, not weeks — the key is
   known-compromised), remove the old key from both secrets; redeploy. Outstanding
   old-key reset/magic links die; **the mobile fleet is forced to re-login once** (both
   apps hard-logout on 401, no refresh) — schedule off-peak; no app release needed.
4. **Cookie key:** rotate `SSO_COOKIE_KEYS` the same way (new first, drop `"subzero"`
   after active OIDC interactions expire — minutes).
5. **Ops script:** update/retire `ops-and-extracts/mint-haven-reset-jwt.js` (reads the
   old key from source).
6. **History & non-prod:** scrub key blobs from git history (after rotation) and rotate
   the leaked non-prod keys (qa/sandbox/uat) via the same externalized mechanism.
